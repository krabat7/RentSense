# Диагностика парсера: данные не пишутся в БД

Выполняй на **сервере** в каталоге проекта (где `docker-compose`).

---

## 1. Логи контейнера парсера (последние 200 строк)

```bash
docker compose -f docker-compose.prod.yml logs parser --tail 200
```

Или без `-f` для полного вывода:

```bash
docker compose -f docker-compose.prod.yml logs parser --tail 500
```

**На что смотреть:**
- `Loaded N proxies from .env file` — должно быть **6** (если у тебя 6 прокси).
- `Proxy stats: total=6, banned=X, time_blocked=Y, available=Z` — если **available=0** и **banned=6**, все прокси в временном бане.
- `All proxies blocked for ... minutes` — все по таймеру заблокированы (403/CAPTCHA).
- `No proxies configured in proxyDict` — в `.env` нет PROXY1 или переменные не подхватились.
- `getResponse: ... Status=403` / `blocked for 20 min` — прокси блокируются Цианом.
- `Цикл парсинга #N завершен успешно` — цикл доходит до конца; если после этого нет новых записей в БД, возможны только «EXISTING»/пустые страницы.
- `Error limit 20 reached, stopping` — много ошибок подряд, парсер останавливает цикл.

---

## 2. Файл временных банов прокси

Если **все** прокси попали в временный бан, парсер не сможет делать запросы.

Проверить наличие и содержимое:

```bash
cat .proxy_bans
```

Если файл есть и в нём **6 строк** (по одной на каждый прокси) — все прокси забанены.

**Сброс банов:** удалить файл и перезапустить парсер:

```bash
rm -f .proxy_bans
docker compose -f docker-compose.prod.yml restart parser
```

Через 10–20 секунд снова посмотреть логи:

```bash
docker compose -f docker-compose.prod.yml logs parser --tail 50
```

Должно появиться `Loaded 6 proxies` и при следующем цикле — запросы через прокси.

---

## 3. Проверка, что в .env подхватываются все 6 прокси

Прокси читаются **строго по порядку**: PROXY1, PROXY2, …, обрыв на первом пустом.

```bash
grep -E '^PROXY[0-9]+=' .env | sort -tY -k2 -n
```

Должны быть **PROXY1** … **PROXY6** без пропусков. Если есть только PROXY1–PROXY3 или есть PROXY11 без PROXY10 — лишние не подхватятся.

---

## 4. Лог-файл парсера (если есть)

Часть логов пишется в `rentsense.log` в **рабочей директории контейнера** (при монтировании `./` в `/app` файл может оказаться в корне проекта на сервере):

```bash
tail -100 rentsense.log
```

Если файла нет — смотреть только логи контейнера (п. 1).

---

## 5. Краткая последовательность при «4 дня ничего не пишется»

1. `docker compose -f docker-compose.prod.yml logs parser --tail 300` — есть ли циклы, ошибки, «All proxies blocked», «available=0».
2. `cat .proxy_bans` — не все ли 6 прокси в бане; при необходимости `rm -f .proxy_bans` и `restart parser`.
3. Проверить `.env`: 6 прокси подряд PROXY1…PROXY6.
4. После сброса банов подождать 1–2 цикла (интервал 30 мин) и снова посмотреть логи и наличие новых записей в БД.

Если после сброса `.proxy_bans` и рестарта в логах снова быстро появляются 403 и блокировки по всем прокси — проблема в самих прокси (Циан их режет); тогда менять прокси или их порядок (самый рабочий поставить PROXY1).

---

## Исправление «цикл 2 часа → таймаут → cannot switch to a different thread»

Если в логах повторяется: цикл прерывается по таймауту 120 мин, затем «Не удалось закрыть браузер после таймаута: cannot switch to a different thread», а следующие циклы не показывают getResponse/Insert — браузер остаётся висеть в потоке executor, новые циклы не работают.

**Сделано в коде:** каждый цикл парсинга запускается в **отдельном процессе** (`python -m app.scheduler.run_parser_once`). При таймауте 2 часа процесс **убивается** (SIGKILL), ОС освобождает браузер и память, следующий цикл стартует новым процессом.

После обновления кода на сервере перезапусти парсер: `docker compose -f docker-compose.prod.yml up -d --build parser` (или `restart parser`). В логах должно быть «Парсер запущен: непрерывный режим (цикл в подпроцессе)» и снова появятся getResponse/Insert.
