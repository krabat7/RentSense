# EDA по объявлениям

## Что внутри `eda_v1.ipynb`

- Загрузка из MySQL (те же фильтры, что и в `prepare_data`: flatRent, rent).
- Разрез по `offer_created_at` (когда строка попала к нам) и сравнение с `publication_at`.
- Проверка `price_changes`, сборка `price_actual` как в пайплайне.
- Описательная статистика, гистограммы, категории, пропуски, корреляции.
- Дубликаты по адрес, этаж, площадь, комнаты.
- Два варианта временного разбиения: по `created_at` (как в обучении) и по `publication_at` (30 дней от максимума). Второй на длинном хвосте даёт сдвиг долей train/test.

Запуск: из `ml/eda` нужен `.env` в корне репозитория с переменными `DB_*`. После правок ячеек ноутбук надо выполнить целиком, чтобы снова появились графики в выводе (в git часто хранят без сохранённых outputs).

## Связь с обучением (`ml/prepare_data.py`)

- Целевая цена: `price_actual` из `price_changes`, иначе `offers.price`.
- Фильтры категории и типа сделки как в SQL ноутбука.
- Время: `offer_created_at`, тест — последние 14 дней или последний календарный месяц (см. константы в скрипте).
- Отсев по цене, площади, цене за м², завышенной площади — пороги в `prepare_data`.
- Дубликаты: одна запись с более поздней датой.
- После очистки: `features.add_features_v2`, `interaction_features`, выгрузка train/test в `data/processed`.

Зачем так: `publication_at` и `created_at` по-разному режут время. В проекте split по `created_at`, чтобы тест был позже по моменту появления строки в нашей базе и не тянуть старые публикации вперемешку с новым парсингом.
